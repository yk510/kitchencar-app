# クリダス 本番反映チェックリスト

## 1. 目的

このドキュメントは、`codex/mobile-order-foundation` のような機能ブランチを `main` に取り込み、本番環境へ安全に反映するためのチェックリストです。

GitHub で Pull Request を作成する前提で、事前確認、本番前設定、本番反映後確認までを一連で使えるように整理しています。

---

## 2. 想定フロー

今回の基本フローは以下です。

1. 機能ブランチで開発する
2. ステージング環境で動作確認する
3. GitHub で `main` 向けの PR を作る
4. PR レビュー / 最終確認を行う
5. `main` にマージする
6. 本番 Vercel が `main` をデプロイする
7. 本番で受け入れ確認する

---

## 4. PR 作成前チェック

### 4-1. 開発ブランチの整理

- 反映したい変更がすべて対象ブランチに push 済みである
- 作業途中の不要な差分が残っていない
- `tsconfig.tsbuildinfo` など生成物をコミットしていない
- 最新のステージング確認内容が対象ブランチの状態と一致している

### 4-2. ビルド確認

- ローカルで `npm run build` が成功する
- 型エラーが出ない
- 致命的な lint/format 問題が残っていない

### 4-3. ステージング受け入れ確認

最低限、以下がステージングで通っている。

- ベンダー側ログイン
- 主催者側ログイン
- ベンダー新規登録
- 主催者新規登録
- メール確認コード入力
- ベンダー/主催者の権限制御
- モバイルオーダー注文
- Stripe テスト決済
- LINE 通知
- ベンダー注文ダッシュボード

### 4-4. データベース変更の有無

- SQL 追加/変更があるか把握している
- 本番反映前に流す SQL が整理されている
- 既存データに影響する変更がある場合、影響範囲を説明できる
- ロールバックしにくい変更かどうか把握している

### 4-5. 外部連携の確認

- Supabase 設定が本番でも有効
- LINE Developers の設定内容が本番と整合している
- Stripe の本番/テスト設定を混同していない

---

## 5. PR 作成チェック

### 5-1. PR 基本情報

- Base branch が `main` になっている
- Compare branch が対象ブランチになっている
- PR タイトルが内容を表している
- PR 説明に以下が書かれている

記載推奨項目:

- 何を追加/変更したか
- ステージング確認済み項目
- 必要な SQL
- 必要な環境変数
- 注意点

### 5-2. レビュー観点

- UI変更が期待どおりか
- 権限制御に問題がないか
- 既存導線を壊していないか
- 本番で必要な設定変更が漏れていないか

---

## 6. 本番環境 事前設定チェック

### 6-1. Vercel

- 本番用 Vercel プロジェクトの `Production Branch` が `main` である
- 本番ドメインが Production に正しく紐づいている
- 反映タイミングを把握している

### 6-2. 本番環境変数

以下が本番に正しく設定されている。

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `NEXT_PUBLIC_VENDOR_APP_HOST`
- `NEXT_PUBLIC_ORGANIZER_APP_HOST`
- `NEXT_PUBLIC_VENDOR_AUTH_COOKIE_NAME`
- `NEXT_PUBLIC_ORGANIZER_AUTH_COOKIE_NAME`
- `NEXT_PUBLIC_VENDOR_SUPABASE_STORAGE_KEY`
- `NEXT_PUBLIC_ORGANIZER_SUPABASE_STORAGE_KEY`
- `NEXT_PUBLIC_LINE_LIFF_ID`
- `LINE_MESSAGING_API_CHANNEL_ACCESS_TOKEN`
- `STRIPE_SECRET_KEY`
- `STRIPE_WEBHOOK_SECRET`

### 6-3. Supabase

- 本番 DB に必要な SQL を流す準備ができている
- メールテンプレートが最新方針になっている
  - 新規登録確認はリンク式ではなく確認コード式
- 本番の Redirect URL / Allowed URL が正しい

### 6-4. LINE Developers

- 本番 LIFF の Endpoint URL が本番ドメインになっている
- Messaging API channel と LINE Login channel が同じ Provider 配下である
- 本番 LIFF ID を本番環境変数へ入れている

### 6-5. Stripe

- 本番用 `STRIPE_SECRET_KEY` になっている
- 本番用 Webhook エンドポイントが設定されている
- 本番用 `STRIPE_WEBHOOK_SECRET` が入っている
- テストキーが本番に残っていない

---

## 7. マージ直前チェック

- PR の差分を最終確認した
- レビューコメント対応が完了している
- 重大な未解決不具合がない
- 本番反映タイミングに問題ない
- 必要なら関係者へ反映予定を共有済み

---

## 8. 本番反映手順

### 8-1. マージ前

- 必要な SQL を先に流すか、反映直後に流すか決めている
- 反映順を決めている

### 8-2. マージ

- PR を `main` にマージする
- マージ方法を記録する
  - squash / merge / rebase

### 8-3. デプロイ

- Vercel の本番デプロイ開始を確認する
- デプロイ成功を確認する
- 必要なら `Redeploy` を行う

## 11. 問題発生時の切り分け

問題が起きたら、まず以下を見る。

- Vercel Deployments
- Vercel Runtime Logs
- Supabase Auth / DB / RLS
- LINE Developers 設定
- Stripe Webhook 配信ログ

確認する観点:

- 環境変数不足
- ドメイン違い
- DB 未反映 SQL
- 権限/RLS
- Webhook secret 違い
- Hosted URL / LIFF URL の不一致

---

## 12. ロールバック判断メモ

以下に当てはまる場合はロールバックを検討する。

- ログインできない
- 決済できない
- 注文保存できない
- LINE 通知が全件失敗する
- 主催者/ベンダーの主要導線が壊れている

ロールバック時に必要なもの:

- 直前の安定コミット
- Vercel で戻す対象デプロイ
- 追加 SQL の影響把握
